# 내부 협업 웹 컨셉 정의 보고서
## exploit 분석 자동화 프로젝트를 위한 운영 구조 설계 초안

### 0. 문서 목적
이 문서는 “최종 서비스 기획안”이 아니라, 현재 팀 상황에 맞는 내부 협업 웹의 컨셉을 정리하는 문서다. 목적은 자동화 기능을 바로 설계하는 것이 아니라, 연구 단계에서 실행 단계로 넘어가는 팀이 어떤 운영 구조를 먼저 가져야 하는지를 선명하게 정의하는 데 있다.

이 문서의 전제는 다음과 같다.

- 이 웹은 외부 공개용 제품이 아니라 팀 내부 운영 협업툴이다.
- `/`는 추후 랜딩페이지로 분리 가능하지만, 실사용 중심 진입점은 `/app`이어야 한다.
- Notion은 장기적으로 대체 대상이다.
- Slack은 완전 대체가 아니라 보조 채널로 유지한다.
- 자동화 실행기, GCP 작업 실행, MCP 연동, 보고서 자동 생성은 MVP 이후 과제로 분리한다.

---

## 1. 프로젝트의 본질적 문제 정의

### 1-1. 현재 문제는 “Notion이 불편하다”가 아니다
지금 팀의 문제는 도구의 UI 문제가 아니라, **운영 대상이 다른 정보들이 같은 형식으로 취급되고 있다는 점**이다. 현재 한 공간 안에 아래 요소들이 모두 섞여 있다.

- 논문 정리와 배경 조사
- 회의 내용과 멘토링 피드백
- 실제 실행해야 하는 작업
- 담당자와 진행 상태
- 디컴파일 결과, 링크, 로그, 코드 비교 자료
- 향후 자동화 도구에 반영해야 하는 학습 내용

이 구조에서는 정보는 쌓이지만 운영은 정리되지 않는다.

### 1-2. 왜 혼잡해지는가
혼잡의 원인은 세 가지다.

첫째, **읽는 정보와 실행하는 일이 섞여 있다.**  
논문 요약, 레퍼런스, 회의 메모는 “읽고 이해하기 위한 정보”다. 반면 담당자, 다음 액션, handoff, 진행 상태는 “움직여야 하는 운영 정보”다. 이 둘은 관리 목적이 다르다. 읽는 정보는 많이 연결될수록 좋지만, 실행 정보는 현재 상태와 책임이 명확할수록 좋다.

둘째, **케이스 단위의 추적 구조가 없다.**  
이 프로젝트는 일반적인 문서 작성 프로젝트가 아니라 exploit case를 통해 사실을 확인하고, 산출물을 남기고, 재현과 root cause 분석으로 연결하는 구조다. 그런데 현재는 각 정보가 case를 중심으로 묶이지 않고 흩어져 있다.

셋째, **연구 단계 정보와 제품화 단계 작업이 분리되지 않았다.**  
지금 팀은 단순히 exploit 하나를 분석하는 것이 아니라, 각 케이스를 통해 최종 자동화 도구를 설계하고 최적화하고 있다. 즉, 케이스 분석과 제품 개발이 동시에 돌아간다. 그런데 이 둘이 분리되지 않으면 “지금 분석 중인 일”과 “재사용 가능한 기능으로 일반화해야 하는 일”이 뒤섞인다.

### 1-3. 단일 문서 허브가 감당하지 못하는 이유
단일 문서 허브는 지식 저장에는 유효하지만, 이 프로젝트처럼 **상태 전이, 책임 이동, 산출물 검증, 주간 운영**이 중요한 구조에는 맞지 않는다. 이 프로젝트에서 팀이 매일 확인해야 하는 질문은 아래와 같다.

- 지금 어떤 케이스가 살아 있고 어디까지 왔는가
- 어느 단계가 막혔고 누구에게 공이 있는가
- 어떤 산출물이 검증되었고 어떤 것은 아직 가설인가
- 이번 주에 무엇을 밀어야 하고 멘토링 이후 무엇이 바뀌었는가
- 특정 케이스에서 얻은 학습이 자동화 도구의 어떤 task로 전환되었는가

문서 허브는 이 질문에 일관되게 답하기 어렵다. 따라서 문제는 Notion 사용법이 아니라 **운영 모델 부재**다.

## 2. 이 내부 웹의 핵심 정체성 정의

### 2-1. 이 웹은 무엇이어야 하는가
이 웹의 핵심 정체성은 다음 한 문장으로 정리할 수 있다.

**“exploit 사례를 기반으로 자동화 도구를 만들어 가는 팀의 운영 중심 협업 시스템”**

이 정의에서 중요한 점은 두 가지다.

- 이 웹은 단순 문서 도구가 아니다.
- 이 웹은 아직 실행기 중심 자동화 콘솔도 아니다.

현재 단계에서 가장 맞는 성격은 아래 세 가지의 조합이다.

| 성격 | 필요성 | 우선순위 |
|---|---|---|
| 케이스 관리 시스템 | exploit 단위의 상태와 산출물 추적이 필요함 | 가장 높음 |
| 운영 협업툴 | 역할 분담, handoff, 주간 진행 관리가 필요함 | 높음 |
| 분석 워크스페이스 | artifact와 메모, 결정, 결과를 연결해야 함 | 높음 |
| 자동화 실행 콘솔 | 미래에는 필요하지만 지금 핵심은 아님 | 낮음 |

### 2-2. 지금 단계와 미래 단계의 제품 정체성
지금 단계의 제품 정체성은 **“사람이 협업 구조를 정리하는 툴”**이다.  
미래 단계의 제품 정체성은 **“자동화 단계를 실행하고 결과를 수집·정리하는 툴”**로 확장될 수 있다.

즉, 지금은 다음이 우선이다.

- 무엇을 운영 단위로 볼 것인가
- 어떤 입력이 어느 단계의 출력이 되는가
- 산출물과 상태와 논의를 어떻게 분리할 것인가
- 케이스에서 얻은 학습을 제품 task로 어떻게 끌어올릴 것인가

자동화는 이 구조가 정리된 뒤 붙어야 한다. 그렇지 않으면 자동화는 운영을 정리하는 것이 아니라 혼란을 더 빠르게 증폭시키는 도구가 된다.

## 3. 이 프로젝트를 보는 기본 프레임: “Case 기반 검증 + Product Task 기반 일반화”

이 프로젝트는 단순히 exploit case를 많이 처리하는 프로젝트가 아니다. 반대로 단순히 기능 목록을 쌓아가는 제품 개발 프로젝트도 아니다. 실제 구조는 두 축이 함께 돌아가는 형태다.

### 3-1. Case는 무엇인가
Case는 exploit 하나를 기준으로 사실을 수집하고, 분석을 남기고, 재현 가능성과 원인을 검증하는 **학습과 검증의 단위**다.

Case는 다음 질문에 답한다.

- 이 exploit에서 실제로 무슨 일이 일어났는가
- 어떤 데이터와 코드 근거가 확보되었는가
- 어떤 분석 흐름이 유효했고 어디서 막혔는가
- 어떤 자동화 가능성이 드러났는가

### 3-2. Task는 무엇인가
Task는 단순 체크리스트가 아니라, **케이스를 통해 드러난 문제를 해결하여 최종 자동화 도구를 개선하는 실행 단위**다.

Task는 크게 두 종류로 나뉜다.

| Task 유형 | 의미 | 예시 |
|---|---|---|
| Case Task | 특정 케이스를 더 진행하기 위한 작업 | 특정 tx 데이터 정제, 디컴파일 비교, 재현 시도 |
| Product Task | 케이스에서 반복적으로 드러난 문제를 일반화해 도구에 반영하는 작업 | 디컴파일 결과 표준화, artifact schema 정의, handoff 템플릿 구축 |

### 3-3. 왜 이 구분이 중요한가
이 구분이 없으면 두 가지 문제가 동시에 생긴다.

- 모든 일이 “눈앞의 케이스 처리”로만 흘러서 제품화가 안 된다.
- 반대로 모든 일이 “언젠가 만들 기능”로만 흘러서 실제 검증 근거가 약해진다.

따라서 이 내부 웹은 **case를 중심으로 운영하되, case에서 나온 학습이 product task로 연결되는 구조**를 가져야 한다.  
즉, 이 시스템의 핵심은 단순 task board가 아니라 **case와 productization을 연결하는 운영 보드**다.

## 4. 사용자/역할 관점 정리: 고정 직무가 아니라 유동적 책임 구조

### 4-1. 역할은 고정 조직도가 아니라 “책임 슬롯”으로 봐야 한다
현재 프로젝트는 방향성과 세부 범위가 완전히 고정된 상태가 아니다. 따라서 처음부터 `tx 담당`, `decompile 담당`, `report 담당`처럼 끝까지 고정된 역할 체계를 설계하는 것은 맞지 않다.

대신 이 웹은 **사람의 직무를 고정하는 도구가 아니라, 각 시점에 필요한 책임을 배정하는 도구**여야 한다.

### 4-2. 권장 책임 구조
아래는 고정 직책이 아니라, 케이스와 주차 단위로 유동적으로 배정할 수 있는 책임 슬롯이다.

| 책임 슬롯 | 의미 | 이 웹에서 봐야 하는 것 | 입력해야 하는 것 | 넘겨줘야 하는 것 |
|---|---|---|---|---|
| Case Owner | 특정 케이스의 전체 흐름을 관리 | 케이스 상태, 병목, 연결된 task/artifact | 우선순위, 현재 상태, 다음 단계 기준 | 다음 workstream에 필요한 조건 |
| Workstream Owner | 특정 시점의 분석/구현 흐름 담당 | 본인에게 배정된 입력, 선행 artifact, 관련 discussion | 작업 결과, 산출물, 실패 이유, 필요 지원 | 검토 가능한 artifact 또는 후속 요청 |
| Product Task Owner | 자동화 도구 측면의 일반화 작업 담당 | 연결된 case 학습, backlog 상태, 구현 우선순위 | 기능 정의, 기준, 진행 상태 | 다음 주차나 검토 단계로 갈 작업 결과 |
| Reviewer / Decision Owner | 결과 검토와 기준 확정 담당 | 검토 대기 항목, 확정 필요 사항, 리스크 | 승인/반려, 기준, 의사결정 메모 | 확정 상태와 다음 방향 |
| Weekly Coordinator | 주간 운영과 멘토링 흐름 정리 | 주간 목표, 멘토링 안건, blocker, carry-over | 주간 요약, 안건, 결정 반영 내용 | 다음 주 실행 항목과 합의된 우선순위 |

### 4-3. 역할 설계 원칙
이 구조에서 중요한 것은 역할 이름이 아니라 운영 원리다.

- 한 사람이 여러 책임 슬롯을 동시에 가질 수 있어야 한다.
- 동일한 사람이 매주 같은 역할을 맡는다고 가정하면 안 된다.
- workstream 종류 자체도 고정하지 않는다.
- 케이스마다 필요한 workstream을 다르게 둘 수 있어야 한다.
- 시스템은 “사람 분류”보다 “현재 책임 위치”를 보여줘야 한다.

즉, 이 웹은 사람을 분류하는 툴이 아니라 **책임의 이동과 handoff를 추적하는 툴**이어야 한다.

## 5. 케이스 중심 운영 구조의 필요성

### 5-1. 왜 task 중심만으로는 안 되는가
이 프로젝트의 최종 목표는 자동화 도구지만, 그 도구의 타당성과 구조는 케이스를 통해 검증된다. 따라서 task만 보면 “무엇을 만들고 있는지”는 보여도, “왜 이 기능이 필요한지”와 “어떤 케이스에서 검증되었는지”가 사라진다.

특히 이 프로젝트에서 task는 각 단계의 최적화와 일반화를 위해 존재한다. 즉, task는 목적 자체가 아니라 **case에서 드러난 반복 문제를 해결하기 위한 수단**이다.

### 5-2. 케이스 중심 구조가 필요한 이유
케이스 중심 구조는 다음을 가능하게 한다.

- exploit별로 상태와 산출물을 묶어 관리할 수 있다.
- 각 케이스에서 어떤 학습이 나왔는지 추적할 수 있다.
- product task가 어떤 실제 케이스에 근거하는지 설명할 수 있다.
- 팀이 “기능을 만들고 있는지”와 “실제 문제를 풀고 있는지”를 동시에 확인할 수 있다.

### 5-3. case와 task의 관계
이 둘은 대체 관계가 아니라 연결 관계다.

| 객체 | 질문 | 역할 |
|---|---|---|
| Case | 무엇을 검증하고 있는가 | exploit 단위의 사실·근거·진행 관리 |
| Task | 무엇을 해야 하는가 | 케이스 진행 또는 제품 일반화 작업 |
| Artifact | 무엇이 나왔는가 | 로그, 결과, 링크, 파일, 표, 요약 |
| Discussion | 왜 이렇게 해석했는가 | 의견, 토론, 가설, 피드백 |
| Decision | 무엇을 확정했는가 | 기준, 방향, 우선순위, 멘토링 반영 사항 |
| Report | 무엇을 설명할 것인가 | 최종 전달용 서술 결과 |

### 5-4. 분리가 필요한 이유
artifact, discussion, status, decision이 섞이면 다음 문제가 생긴다.

- 결과물과 가설을 구분할 수 없다.
- “검토 중”과 “확정됨”을 구분할 수 없다.
- 멘토링 피드백이 실제 task로 반영되었는지 추적할 수 없다.
- 하나의 case에서 나온 학습이 제품 과제로 승격되었는지 보이지 않는다.

따라서 케이스 중심 구조는 단순 정리 방식이 아니라, **연구와 제품화를 연결하는 최소 운영 단위**다.

## 6. 내부 협업툴로서 필요한 정보 구조(IA) 컨셉

이 웹은 문서를 많이 쌓는 시스템이 아니라, 서로 다른 운영 객체를 명확히 분리하는 시스템이어야 한다. 핵심 정보 단위는 아래와 같다.

| 정보 단위 | 역할 | 왜 필요한가 |
|---|---|---|
| Case | exploit 단위 운영의 기준 객체 | 모든 상태와 산출물을 묶는 중심 축 |
| Case Task | 특정 케이스를 진행하기 위한 작업 | 현재 액션과 담당자 추적 |
| Product Task | 케이스 학습을 제품화하기 위한 작업 | 자동화 도구 개발 backlog 관리 |
| Artifact | 실제 결과물 저장 | 링크, 파일, 로그, 비교표, 결과 스냅샷 관리 |
| Discussion | 맥락 있는 논의 | Slack에 흩어질 해석과 토론을 케이스 안에 남김 |
| Decision | 확정된 판단 | 방향성 변경, 멘토링 반영, 기준 확정 추적 |
| Weekly Cycle | 주간 운영 단위 | 주차별 목표와 진행률 관리 |
| Mentoring Note | 멘토링 기록 단위 | 멘토 피드백을 주간 운영과 연결 |
| Report | 최종 정리본 또는 초안 | 나중의 보고 자동화 전 단계 결과 |
| Event Log | 타임라인 기록 | 누가 언제 무엇을 바꿨는지 추적 |

### 6-1. Notion 구조에서 문제가 된 이유
기존 Notion 구조에서는 위 객체들이 대부분 “페이지”나 “문단”처럼 동일 레벨로 취급된다. 그 결과 아래 문제가 생긴다.

- case와 주간 회의 메모가 같은 성격처럼 보인다.
- task와 decision이 같은 형태로 쌓인다.
- artifact가 메모 속 링크로만 남아 재사용이 어렵다.
- 멘토링 피드백이 실제 action으로 전환되지 않는다.

따라서 새 내부 웹은 문서를 예쁘게 정리하는 것이 아니라, **운영 객체를 분리하고 관계를 명확히 드러내는 IA**를 가져야 한다.

## 7. 운영 흐름 컨셉: exploit 후보부터 보고서 직전까지

이 툴의 운영 흐름은 단순 기능 순서가 아니라, **입력과 출력이 명확한 handoff 흐름**이어야 한다. 다만 세부 분석 단계는 앞으로 바뀔 수 있으므로, coarse state는 고정하고 세부 workstream은 유동적으로 설계하는 것이 적절하다.

### 7-1. 권장 coarse flow
| 단계 | 입력 | 출력 |
|---|---|---|
| Intake | exploit 후보 정보, tx hash, 초기 메모 | 관리 가능한 신규 case |
| Triage | 기본 메타데이터, 위험도, 조사 가치 | 우선순위와 초기 scope |
| Workstream Setup | triage 결과, 필요한 조사 방향 | 연결된 case task와 담당 책임 |
| Evidence Build | 데이터 추출, 디컴파일, 비교, 메모 | 검토 가능한 artifact 묶음 |
| Validation | 재현 시도, root cause 검증, 반례 확인 | 확정 가능한 근거와 decision |
| Synthesis | 확정된 근거, 핵심 관찰 | 보고용 구조화된 summary/report 초안 |
| Productization Feed | case 학습과 반복 패턴 | product task 또는 backlog 반영 |

### 7-2. 중요한 운영 원리
- 모든 단계는 “완료”가 아니라 “다음 단계가 쓸 수 있는 출력”으로 정의해야 한다.
- 세부 workstream은 case별로 달라질 수 있어야 한다.
- case 상태와 product task 상태를 분리해서 보되, 링크로 연결해야 한다.
- 멘토링 이후 바뀐 우선순위나 기준은 decision으로 남아야 한다.
- Slack에서 논의된 내용도 결국 case나 weekly cycle에 귀속되어야 한다.

이 흐름은 “exploit 하나를 끝내는 과정”이면서 동시에 “자동화 도구의 요구사항을 정제하는 과정”이기도 하다.

## 8. 주간 운영 및 멘토링 추적 구조

주차별 멘토링이 있다는 점은 이 프로젝트 운영 구조에서 매우 중요하다. 이유는 멘토링이 단순 회의가 아니라, **우선순위와 방향 수정이 정기적으로 발생하는 의사결정 이벤트**이기 때문이다.

### 8-1. 주간 운영 단위가 필요한 이유
케이스만 보면 개별 분석은 보이지만, “이번 주에 팀이 어디에 집중했는가”는 잘 안 보인다. 반대로 task만 보면 일은 보이지만, 왜 그 주에 그 일을 했는지 맥락이 사라진다. 따라서 주차 단위의 운영 객체가 별도로 필요하다.

### 8-2. 주간 구조에서 관리해야 할 정보
| 주간 객체 | 역할 |
|---|---|
| Weekly Goal | 이번 주 집중 목표 |
| Active Cases | 이번 주 집중하는 case 목록 |
| Active Product Tasks | 이번 주 밀어야 할 제품화 작업 |
| Blockers / Risks | 현재 막힌 점과 의사결정 필요 사항 |
| Mentoring Agenda | 멘토링 때 확인받아야 할 쟁점 |
| Mentoring Feedback | 받은 피드백 원문 또는 요약 |
| Weekly Decisions | 멘토링 후 확정된 방향 변화 |
| Carry-over | 다음 주로 넘기는 항목 |

### 8-3. 운영 흐름상 주간 화면이 답해야 하는 질문
- 이번 주 팀이 무엇에 집중하는가
- 케이스 기준으로 어디까지 왔는가
- 제품 기준으로 어떤 일반화 작업이 진행 중인가
- 멘토링에서 무엇을 물을 것인가
- 멘토링 이후 무엇이 바뀌었는가
- 다음 주에 어떤 항목이 이어지는가

### 8-4. 멘토링 추적의 핵심 원칙
- 멘토링은 회의록 저장이 아니라 decision capture여야 한다.
- 멘토 피드백은 case 또는 product task와 연결되어야 한다.
- 주간 정리는 “무엇을 했는가”보다 “무엇이 바뀌었는가”가 중심이어야 한다.
- 주차별 운영은 리포트용 정리가 아니라 실제 우선순위 조정 도구여야 한다.

## 9. Slack / Notion / 내부 웹의 역할 분담

### 9-1. Notion에 대한 입장
Notion은 이 구조에서 **장기적으로 완전 대체 대상**으로 보는 것이 맞다. 다만 전환기에는 기존 자료를 참조할 수 있으므로 읽기 전용 레퍼런스 소스로 잠시 남을 수 있다. 그러나 운영 설계의 기준은 “Notion이 없다”를 전제로 해야 한다.

중요한 점은 Notion을 대체한다고 해서, 범용 문서 에디터를 새로 만드는 것이 아니라는 점이다. 내부 웹이 가져야 하는 것은 Notion의 자유로운 문서성이 아니라, **case/weekly/decision 중심 구조화된 기록성**이다.

### 9-2. Slack에 대한 입장
Slack은 완전 대체 대상이 아니다. Slack은 여전히 유효한 보조 채널이다. 다만 역할은 분명히 제한해야 한다.

| Slack에 남길 것 | 이유 |
|---|---|
| 빠른 조율 | 실시간 커뮤니케이션에 적합 |
| blocker 알림 | 반응 속도가 중요함 |
| 리뷰 요청 | 사람을 깨우고 흐름을 당기는 용도 |
| 멘토링 전후 공지 | 일정성, 즉시성에 유리 |
| 요약 브로드캐스트 | 주간 진행/변경 사항 공유에 적합 |

| Slack에 남기지 말 것 | 이유 |
|---|---|
| 케이스의 공식 상태 | 스레드가 흩어지고 검색성이 낮음 |
| 최종 의사결정 원본 | 근거와 연결이 끊김 |
| artifact의 기준 링크 저장소 | 맥락 없이 링크만 떠다님 |
| task ownership의 공식 기준 | 누가 현재 책임자인지 불명확해짐 |

### 9-3. Slack 중심으로 다시 정리하면
Slack은 운영의 중심이 아니라 **전달과 반응의 표면**이다.  
내부 웹은 운영의 중심이자 source of truth다.

즉, 권장 구조는 다음과 같다.

- 내부 웹: 상태, 책임, 산출물, 결정의 기준점
- Slack: 알림, 빠른 조율, 리뷰 요청, 주간 공지
- Notion: 점진적 종료 또는 읽기 전용 참조

이 프로젝트에서 Slack을 계속 쓰더라도, 규칙은 명확해야 한다.  
**Slack에서 결정된 것은 결국 내부 웹에 귀속되어야 한다.**

## 10. 1차 제품 범위(MVP)와 Post-MVP 자동화 과제 구분

### 10-1. 지금 반드시 필요한 것
| 항목 | 판단 | 이유 |
|---|---|---|
| Case 생성/조회 | 지금 필요 | 운영의 기본 단위 |
| coarse 상태 관리 | 지금 필요 | 병목과 handoff 가시화 |
| 책임 배정 | 지금 필요 | 유동적 역할 운영의 기반 |
| Case Task / Product Task 분리 | 지금 필요 | 케이스 진행과 제품화 작업을 분리해야 함 |
| Artifact 첨부/링크 | 지금 필요 | 산출물을 케이스 맥락에 귀속시켜야 함 |
| Discussion / Decision 분리 기록 | 지금 필요 | 가설과 확정을 구분해야 함 |
| 주간 운영 보드 | 지금 필요 | 멘토링/주차 운영 중심 관리 필요 |
| Mentoring note 연결 | 지금 필요 | 주간 피드백을 실제 action으로 연결해야 함 |
| 기본 검색/필터 | 지금 필요 | 케이스, 담당, 주차 기준 탐색 필요 |

### 10-2. 곧 필요하지만 MVP에서 과하게 만들 필요는 없는 것
| 항목 | 판단 | 이유 |
|---|---|---|
| 보고서 초안 템플릿 | 곧 필요 | 최종 산출물 연결은 중요하지만 자동화까진 이르다 |
| cross-case 패턴 뷰 | 곧 필요 | 자동화 요구사항 도출에 유용 |
| Slack 알림 연동 | 곧 필요 | 운영 속도를 높이지만 핵심 모델이 먼저다 |
| 기존 Notion 자료 마이그레이션 보조 | 곧 필요 | 전환 비용을 줄일 수 있음 |

### 10-3. MVP 이후로 미뤄야 하는 것
| 항목 | 판단 | 이유 |
|---|---|---|
| 자동 실행 버튼 | Post-MVP | 운영 구조 없이 실행기부터 붙이면 예외만 늘어남 |
| GCP 잡 실행 | Post-MVP | 입력/출력 계약과 책임 구조가 먼저다 |
| MCP 연결 | Post-MVP | 연동 포인트는 모델 안정화 후 붙여야 한다 |
| 디컴파일 파이프라인 오케스트레이션 | Post-MVP | 실행 자동화는 범위가 크고 리스크가 높다 |
| 보고서 자동 생성 | Post-MVP | 보고 구조와 artifact schema가 먼저 정리돼야 한다 |
| 외부 공개용 랜딩페이지 | Post-MVP | 내부 운영 속도와 무관하다 |

### 10-4. 핵심 판단
MVP의 목표는 “분석 자동화”가 아니다.  
MVP의 목표는 **“case와 product task와 주간 운영을 한 구조 안에서 관리할 수 있게 만드는 것”**이다.

## 11. 화면 철학 및 UX 방향성

### 11-1. 이 제품의 화면 성격
이 프로젝트의 화면은 마케팅 랜딩형이 아니라 **운영 대시보드형**이어야 한다. `/app`이 첫 실사용 화면이라는 전제를 기준으로 보면, 가장 맞는 느낌은 단순 “관제실”보다는 **analysis workspace + operation board**의 조합이다.

“incident room”의 긴장감은 일부 맞지만, 이 팀은 실시간 대응만 하는 조직이 아니다. 논문 기반 리서치, 검증, 제품화가 함께 돌아가기 때문에 단순 incident UI는 부족하다. 반대로 pure document workspace도 맞지 않는다. 가장 적절한 방향은 아래와 같다.

- 중심 느낌: analysis workspace
- 보조 성격: operation board
- 필요한 긴장감: incident room 일부 차용
- 피해야 할 방향: 마케팅 서비스 화면, 범용 문서 툴, 예쁜 kanban 장난감

### 11-2. UX 원칙
- 예쁜 화면보다 상태 가시성이 우선이다.
- 한 화면에 모든 정보를 몰아넣지 않는다.
- case를 기준으로 상태, artifact, discussion, task가 계층적으로 보여야 한다.
- 역할별로 필요한 정보량이 달라야 한다.
- 주간 운영 화면과 케이스 상세 화면은 분리하되 연결돼야 한다.
- 빠른 기록보다 구조화된 기록이 우선이다.
- 자유 메모를 허용하더라도 반드시 case 또는 weekly context에 귀속시켜야 한다.

### 11-3. 첫 화면이 보여줘야 하는 것
`/app` 첫 화면은 아래 질문에 답해야 한다.

- 지금 활성 케이스는 무엇인가
- 어디가 막혔는가
- 누가 무엇을 들고 있는가
- 이번 주 우선순위는 무엇인가
- 멘토링 전후로 무엇이 바뀌었는가

이 질문에 답하지 못하면, 화면이 아무리 보기 좋아도 내부 협업툴로서 실패다.

## 12. 냉정한 평가

### 12-1. 이 방향의 장점
- 연구 단계와 실행 단계의 경계를 정리할 수 있다.
- case에서 나온 학습을 제품 task로 연결할 수 있다.
- 주차별 운영과 멘토링 반영이 쉬워진다.
- “누가 지금 무엇을 해야 하는가”가 명확해진다.
- Notion 문서 더미에서 벗어나 실제 운영의 기준점을 만들 수 있다.

### 12-2. 이 방향의 리스크
- 협업툴을 직접 만들다가 내부 플랫폼 프로젝트로 커질 수 있다.
- Notion을 완전 대체하려는 순간 범용 문서 기능까지 욕심낼 위험이 있다.
- 팀 규모 7명에 비해 너무 정교한 권한/상태/워크플로우를 설계하면 과설계가 된다.
- 자동화 욕심을 초기에 같이 넣으면 구조 설계보다 실행기 디버깅에 시간을 쓰게 된다.
- Slack을 계속 쓰면서 기록 규율이 없으면 결국 source of truth가 다시 분산된다.

### 12-3. 무엇을 포기해야 빨라지는가
이 프로젝트를 빠르게 전진시키려면 아래를 포기해야 한다.

- 외부 공개용 랜딩페이지
- 화려한 브랜딩 중심 UI
- 범용 문서 에디터 재구현
- 복잡한 권한 체계
- 초반 GCP 실행기 탑재
- MCP 연동 욕심
- 자동 보고서 생성까지 한 번에 넣는 설계

포기해야 하는 이유는 단순하다. 지금 팀이 모자란 것은 기능이 아니라 **운영 구조**이기 때문이다.

### 12-4. 현실적인 결론
지금 내부 웹을 먼저 만드는 방향은 맞다. 다만 전제가 있다.  
그 웹은 “뭔가 많이 되는 웹”이 아니라, **case, task, artifact, weekly decision을 분리해 보여주는 최소 운영 시스템**이어야 한다.

## 13. 최종 정리

### 13-1. 이 내부 웹의 한 줄 정의
**exploit 사례를 통해 자동화 도구를 만들어 가는 팀의 케이스 중심 운영 협업 시스템**

### 13-2. 지금 꼭 잡아야 할 핵심 원칙
- 운영의 기준 객체는 문서가 아니라 `Case`여야 한다.
- `Case`와 `Product Task`를 분리하되 연결해야 한다.
- `Artifact`, `Discussion`, `Decision`, `Status`를 섞지 말아야 한다.
- 역할은 고정 직무가 아니라 유동적인 책임 슬롯으로 관리해야 한다.
- 주간 운영과 멘토링 반영은 별도 객체로 추적해야 한다.
- 내부 웹이 source of truth가 되고 Slack은 보조 채널이어야 한다.
- 자동화는 MVP 이후 과제로 분리해야 한다.

### 13-3. 실제 기획안으로 넘어갈 때 가장 먼저 구체화해야 할 항목
- `Case / Case Task / Product Task / Artifact / Discussion / Decision / Weekly Cycle / Mentoring Note`의 데이터 모델
- coarse case state와 handoff 규칙
- `/app` 첫 화면의 운영 대시보드 구조
- 케이스 상세 화면에서 보여줄 최소 정보 계층
- 주간 운영 보드와 멘토링 반영 흐름
- Slack 알림이 어떤 이벤트를 기준으로 발생할지에 대한 운영 규칙

이 문서를 기준으로 다음 단계에서는 바로 `MVP 페이지 구조`, `핵심 사용자 시나리오`, `case state 설계`, `weekly 운영 보드 구조`로 이어서 구체화하는 것이 맞다.
